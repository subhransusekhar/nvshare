# nvshare v0.2 ‚Äî M0 production-hardening

## Highlights

- **Modern CUDA VMM hook coverage** ‚Äî hooks the CUDA Virtual Memory Management
  family (`cuMemCreate`, `cuMemAddressReserve`, `cuMemMap`, `cuMemRelease`,
  `cuMemSetAccess`). Fixes silent bypass on PyTorch with
  `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True`. (#1)
- **Stream-ordered allocator hooks** ‚Äî hooks the async-allocator family
  (`cuMemAllocAsync`, `cuMemFreeAsync`, `cuMemPoolCreate`,
  `cuMemPoolSetAttribute`). Covers PyTorch 2.0+ async memory paths. (#2)
- **Unified Memory hint hooks** ‚Äî hooks `cuMemPrefetchAsync`,
  `cuMemAdvise`, and `cuMemRangeGetAttribute` so prefetch hints reach the
  managed-memory backing. (#3)
- **Single-source-of-truth dispatch in `cuGetProcAddress`** ‚Äî all CUDA 12+
  symbol lookups go through `hook_proc_addr.c` dispatch; eliminates the dual
  maintenance burden and the bypass risk when the CUDA Runtime uses
  `cuGetProcAddress` instead of the ELF symbol table. (#4)
- **`cuGetProcAddress_v2` coverage + CUDA 12 smoke test** ‚Äî audited and
  hooked `cuGetProcAddress_v2`; added a CI smoke test on a CUDA 12 image
  to catch regressions. (#5)
- **CUDA matrix CI** ‚Äî fast PR check plus a nightly 5 CUDA √ó 3 Ubuntu LTS
  (20.04 / 22.04 / 24.04) matrix build. (#6)
- **Dual-glibc artifacts** ‚Äî ships two builds (glibc 2.31 and glibc 2.34)
  covering Ubuntu 20.04 / 22.04 / 24.04 LTS, Debian 11 / 12, and RHEL 9.
  `scripts/select-libnvshare.sh` picks the right one at runtime. (#7)
- **Cosign-signed multi-arch container images** ‚Äî core images published to
  GHCR (`ghcr.io/subhransusekhar/nvshare`) and signed keyless with cosign
  on every push. (#8)
- **Scheduler state persistence** ‚Äî scheduler writes
  `/var/run/nvshare/state.json` on every lock-cycle transition; survives
  `SIGKILL` and restarts with no client disruption. (#9)
- **Client reconnect with cached ID** ‚Äî clients retry the scheduler socket
  on connection loss and reuse their persisted client ID; reconnect within
  ~5 s of the scheduler coming back. (#10)
- **HEARTBEAT protocol** ‚Äî new message type; scheduler sweeps for missing
  heartbeats every 30 s and reaps zombie clients, preventing lock starvation
  after a client crash. (#11)
- **Fail-soft path verified** ‚Äî smoke-tested the `scheduler unreachable`
  path: workload continues (UM oversubscription active), anti-thrash silently
  disabled. Documented in `docs/runbook.md`. (#12)
- **Structured JSON logging** ‚Äî `NVSHARE_LOG_FORMAT=json` emits one-line
  JSON objects with `level`, `event`, and `msg` fields; all lock-cycle
  events migrated to `log_event()`. (#13)
- **Prometheus `/metrics` on `:9601`** ‚Äî five core metrics:
  `clients_connected`, `lock_acquisitions_total`, `lock_wait_seconds`,
  `lock_hold_seconds`, `early_releases_total`. (#14)
- **Per-client request IDs** ‚Äî each `REQ_LOCK` carries a 16-hex-char
  `req_id` propagated through every scheduler log event, enabling
  end-to-end tracing of a single lock cycle. (#15)
- **pytorch-vmm test workload** ‚Äî `tests/pytorch-vmm.py` exercises the
  VMM hook path with `expandable_segments:True`; included in the CI matrix.
  (#16)
- **Scheduler-restart soak test** ‚Äî `tests/scripts/soak-restart.sh` kills
  the scheduler with `SIGKILL` at a configurable interval and asserts zero
  client crashes over the full run duration. (#17)

## Breaking changes

- **`data` field reused for `req_id`.** The 16-byte `data` field of the
  protocol message is now populated with the `req_id` (16 hex chars) on
  both `REQ_LOCK` and `LOCK_OK` messages. Old v0.1 clients interoperate:
  the scheduler accepts their messages and logs `req_id=0000000000000000`.
  No functional break; log correlation requires v0.2 clients.

- **K8s device resource renamed.** The Kubernetes extended resource is now
  `netgroup.ai/gpu` (was `nvshare.com/gpu`). Update pod specs and resource
  quotas before upgrading. This change was made in the first commit on
  `production-roadmap` prior to M0 issue work.

## Known limitations (M0)

- **Single GPU per node.** `nvshare-scheduler` is hardcoded to GPU index 0;
  multi-GPU support is planned for M1.
- **FCFS scheduling only.** No priority classes or weighted fairness; that
  is M2 scope.
- **No Kubernetes operator or CRDs.** DaemonSet-only deployment; a
  controller/operator is M3 scope.
- **`struct nvshare_client` layout coupling.** `scheduler.c` and
  `state_persist.c` share the same struct definition inline. The layout is
  documented and stable for M0; it will be extracted to a shared header in
  M1.

## Upgrade

Drop-in replacement for v0.1 deployments. All new env vars (`NVSHARE_LOG_FORMAT`,
`NVSHARE_ENABLE_SINGLE_OVERSUB`) are opt-in with no-op defaults. A rolling
restart of the `nvshare-scheduler` and `nvshare-device-plugin` DaemonSets is
sufficient; no manifest changes are required for existing workloads unless you
are using `nvshare.com/gpu` in pod specs (update to `netgroup.ai/gpu`).
